Module documentation

Provenance

Append-only event records that preserve who acted, what happened, what inputs and outputs were involved, and which protocol or workflow context framed the event.

Scope and ownership

Provenance is the local ledger surface for reconstructable scientific events. It records immutable events, not mutable current state, and it does so without claiming execution authority, persistence policy, or host behavior. That is an important distinction: the module is concerned with preserving evidence of activity, not with deciding whether the activity was scientifically legitimate in the first place.

A recorded event can carry actor, activity, subject, timestamp, ordered inputs and outputs, and optional protocol or workflow binding. Taken together, those fields make the module the repository’s main answer to “what happened and what did it touch?” while still keeping broader workflow engines or application storage out of scope.

Gate and ADR status

ADR 0008 is the governing contract, and the accepted local implementation scope is recorded in GATE-05 with supporting evidence in GATE-05-EVIDENCE. The current contract is intentionally local: it defines event records and append-only in-memory ordering, not broader enterprise logging or distributed audit infrastructure.

The key fixed points are the nexus.provenance-event schema identity, the provenance-event digest scope, and the stable rejection categories for duplicate IDs, missing actor or required IO, and non-canonical projection kinds. The module also keeps the important governance boundary that plugin, automation, import, or system actors may record provenance but do not become scientific approval authority by doing so.

Code surface and behavior

The main implementation files are ProvenanceEvent.cs, ResearchEventFactory.cs, and ProvenanceStore.cs. The code is compact and readable: a factory validates required actor and IO constraints, an event type carries the digest-bearing record, and the in-memory store enforces append-only ordering with duplicate rejection.

The most important behavior detail is that subject, input, and output references are not allowed to degrade into projection-like identities. Cache rows, wiki pages, local paths, and similar convenience representations are rejected as canonical provenance entities. This is strict, but it is consistent with the project’s broader rule that convenient application state is not the same thing as scientific record.

Verification and current limits

Verification is split cleanly between direct behavior and fixture replay. ProvenanceTests.cs covers the native rules, while ProvenanceFixtureTests.cs replays the contract fixtures under fixtures/conformance/provenance. Architecture tests keep the project inward-facing and free from inappropriate outward dependencies.

The limits are explicit. There is no persistence schema, no compaction or retention strategy, no envelope chain for tamper evidence, and no CLI, UI, or API import/export surface yet. The store is intentionally minimal. That is acceptable for the current gate, but it also means readers should not mistake this module for a complete audit subsystem beyond the accepted local scope.
