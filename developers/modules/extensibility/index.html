Extensibility

Explicit extension identity and capability records for plugin-style workflows, designed to keep execution, runtime, and credentials outside Core domain authority.

Scope and ownership

Extensibility is a narrow contract module. It defines how third-party extensions are described and how authority is narrowed to an explicit capability set. It does not execute extensions, mediate hosts, transport secrets, or provide isolation boundaries by itself.

  • Extension identity contract: Id, Version, EntryPoint, and requested capabilities are treated as explicit, non-runtime metadata.
  • Capability contract: explicit enum for extension rights over protocol/workspace artifacts in a fixed, strongly typed set.
  • Grant contract: selected capabilities are represented as snapshots with ExtensionId plus capability set.
  • Runtime policy is delegated: host, sandboxing, credentials, and transport controls are outside this module.

Gate and ADR status

There is no dedicated accepted gate document for Extensibility yet. It remains in planning/provenance-mapped state from GATE-00, with scope planning in PORT-MATRIX and fixture intent in GOLDEN-FIXTURE-PLAN.

  • Conflict baseline: CF-005 blocks blueprint-to-contract authority unless schema and contract references are reconciled.
  • Discovery blocker: BLUEPRINT-AUDIT flags plugin contract drift; PluginContracts does not match required blueprint execution/data-handling shape yet.
  • Implementation blockers: no plugin runtime, no plugin credential semantics, and no staged execution policy is implemented in this contract module.

Core contracts

  • ExtensionManifest (src/NexusScholar.Extensibility/ExtensionManifest.cs)
  • Factory guard behavior:
    • Id, Version, EntryPoint are required and validated as non-blank via Guard.NotBlank.
    • Capabilities are converted to a HashSet for deterministic duplicate collapse.
  • ExtensionCapability enum (PluginContracts.cs):
    • ReadProtocol
    • ReadArtifacts
    • WriteStagedArtifacts
    • RenderExport
  • CapabilitySelection (CapabilitySelection.cs) as staged grant projection.
  • Module AGENTS rule: do not treat dependency isolation as security and validate staged outputs before they become canonical.

Integration behavior in current code

  • Workflow capability references are declared in WorkflowTemplateCapabilityRequirement as references and optional plugin capability labels, but extension manifests are not used as a runtime grant resolver in Core.
  • Validation impact currently present in Gate-4 workflow behavior: unknown capability refs are rejected at template compile time by ValidateCapabilityRefs.
  • CLI includes project reference to Extensibility for alignment, not runtime extension host behavior.
  • No execution policy, data-handling policy, secrets policy, or plugin invocation pipeline is modeled in this module.

Related modules

  • Workflow: declares capability requirements and plugin capability labels in template nodes.
  • Protocol: extension manifests may eventually reference protocol read capabilities.
  • Artifacts: ReadArtifacts/WriteStagedArtifacts capabilities currently descriptive only, not enforced.
  • AI: AiTaskPolicy introduces another capability-authorization domain with separate ADRs and future gate work; both domains currently do not interlock at runtime.
  • CLI: currently references the project for build alignment only.

Known issues and risks (current)

  • Scope is contract-only: no plugin runtime, no host boundary, and no credential transport are implemented.
  • Blueprint drift remains unresolved: manifest schema and runtime policy fields described in discovery are not yet represented in current local contracts.
  • Minimal capability vocabulary: enum entries are fixed and cannot encode richer per-plugin scopes, egress constraints, trust classes, or data-handling policy required for production integration.
  • No evidence fixtures: PORT-MATRIX/GOLDEN-FIXTURE-PLAN list planned files (`plugin-manifest-minimal.json`, `plugin-permission-grant.json`) but they are not yet present in `fixtures/conformance`.
