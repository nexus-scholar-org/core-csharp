Kernel

Deterministic primitives and canonicalization rules that every higher-level module relies on when the system needs one stable answer for identity, hashing, and time.

Scope and ownership

Kernel is where the repository fixes the low-level rules that cannot drift from module to module. It owns canonical IDs, digest rendering, timestamp normalization, canonical JSON, and envelope rules. That means every domain module can reuse the same vocabulary for content identity without importing application concerns, persistence APIs, or host frameworks.

In practical terms, the module centers on three clusters of types: primitive guards and IDs in KernelPrimitives.cs, canonical content machinery in CanonicalJson.cs and CanonicalTimestamp.cs, and digest contracts in ContentDigest, DigestEnvelope, DigestScope, and DigestAlgorithm.

Gate and ADR status

GATE-02 is the accepted scope for kernel determinism, and ADR 0002 is the decision that makes those rules authoritative. The page-level takeaway is simple: if another module computes a digest, normalizes a timestamp, or serializes canonical scientific content, it should be using Kernel rather than re-inventing the rule locally.

The important fixed points are the lowercase sha256:<64 hex> digest format, the repository’s RFC-8785-style canonical JSON profile, the canonical UTC timestamp shape yyyy-MM-ddTHH:mm:ss.fffffffZ, and the shared digest-scope vocabulary that later modules reuse for protocol content, approval records, provenance events, manifests, and NDJSON streams.

Evidence for the accepted surface lives in GATE-02-EVIDENCE. That matters because this module is foundational: a quiet change here would ripple into nearly every other digest-bearing contract.

Code surface and verification

The implementation is compact and intentionally concentrated. The main files are KernelPrimitives.cs, DigestTypes.cs, ContentDigest.cs, DigestEnvelope.cs, CanonicalJson.cs, CanonicalTimestamp.cs, and NdjsonCanonicalizer.cs.

Verification is equally direct: DeterministicKernelTests.cs exercises the native behavior, while KernelFixtureTests.cs replays the conformance fixtures under fixtures/conformance/kernel/. That combination is appropriate for a low-level module: fast direct tests for the implementation, fixture replay for the contract surface.

Downstream impact and current risks

Kernel is effectively consumed by the rest of the domain stack: Protocol for content and approval digests, Workflow for canonical compile outputs, Provenance for event hashes, Bundles for manifest identity, and the later Search, Screening, Deduplication, AI, Shared, Artifacts, CLI, and Extensibility surfaces for various identity and payload rules.

The current risks are not in coverage volume but in boundary discipline. First, parts of Protocol still carry legacy provisional digest assumptions, so Gate 2 deliberately stops short of claiming protocol-wide cleanup. Second, very large non-integer numeric inputs still depend on explicit parse paths in CanonicalJson.ParseNumber; if another module needs strict precision guarantees beyond that, it must enforce them at its own schema boundary rather than assuming Kernel has solved the policy question for every caller.
