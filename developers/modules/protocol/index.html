Protocol

Scientific protocol records, approvals, waivers, amendments, and immutable approved versions that define what the project is authorized to do before Workflow ever compiles a plan.

Scope and ownership

Protocol owns the lifecycle of scientific intent inside Core. It is where draft records are shaped, where required decisions become explicit, and where approved versions become immutable scientific state. The module depends only on NexusScholar.Kernel, which is the right boundary: protocol meaning stays in the domain layer and does not mix with storage, UI, CLI presentation, or transport concerns.

That lifecycle is more than simple state switching. The code enforces that decision keys are not silently overwritten, approval binds to exact content digests, and approved versions are treated as fixed content rather than mutable working notes. In this repository, Protocol is the authority for what has been approved; other modules only consume that result.

Gate and ADR status

GATE-03 is the accepted local scope for protocol lifecycle behavior, with ADR 0003 defining the record contract and ADR 0004 defining approval semantics. The important reading is that Protocol is already on an accepted contract, not just a scaffold.

The core fixed points are the protocol-content digest for approved content, the approval-record digest for approval snapshots, and the rule that approval authority is human-only. Stale approvals, duplicate approvals, unresolved blocking decisions, and dual-policy failures are rejected with stable categories rather than left to application code to interpret later.

The evidence bundle for this module lives in GATE-03-EVIDENCE. That file matters because it marks the difference between accepted local behavior and any later blueprint or PHP parity claim, neither of which is being asserted here.

Code surface and verification

The public surface is small but consequential: ProtocolDraft.cs, ProtocolModels.cs, and the module project file together describe the working contract. These files define draft creation, decision recording, approval candidates, approved versions, waivers, deviations, and related identity material.

Verification is stronger than the size of the module might suggest. ProtocolTests.cs covers the direct behavior, ProtocolFixtureTests.cs replays the fixture contract under fixtures/conformance/protocol, and WorkflowCompilerTests.cs proves that downstream workflow compilation respects approved protocol invariants rather than bypassing them.

Consumers and current risks

Protocol is consumed directly by Workflow, Bundles, Provenance, and the sample CLI paths. Workflow compiles from approved versions, Bundles bind protocol versions into portable review manifests, Provenance carries protocol binding metadata, and the CLI exercises the path in smoke-style sample behavior. AI is relevant mostly as a non-consumer boundary: automation can propose, warn, or validate, but it does not inherit protocol approval authority.

The current issues are narrow but worth calling out. The convenience overload ProtocolDraft.Create(ids, title, requiredDecisionKeys) still injects hard-coded legacy placeholders, so it should be treated as sample or utility surface rather than a strong domain entry point. The draft identity surface is also a little confusing because ProtocolDraft.Id mirrors ProtocolId instead of cleanly foregrounding DraftId. Finally, withdrawal behavior is only partially surfaced at the contract level; the rules exist, but there is not yet a richer high-level workflow around them.
