Workflow

Deterministic workflow compilation from approved protocol versions into reproducible planned graphs with gate, waiver, and invalidation planning semantics.

Scope and ownership

Workflow turns approved protocol authority into a planned execution graph, without implementing execution, persistence, UI, or plugin host behavior. It is a domain module that depends on Kernel and Protocol and emits canonicalized workflow definitions.

  • Input contract: WorkflowCompileInput with approved protocol version + schema-closed template + optional amendment/notice material.
  • Compile output: WorkflowDefinition containing deterministic node graph, resolved input bindings, approval/capability declarations, artifact declarations, and optional invalidation plan.
  • Authority boundary: protocol approval is authoritative for protocol versions; workflow approval is compiled planning metadata only.

Gate and ADR status

GATE-04 is implemented for local deterministic workflow compiler scope. Core decisions are ADR 0005 and ADR 0006.

  • Compile is explicit-only: WorkflowCompiler.Compile(ProtocolVersion) is rejected in favor of Compile(WorkflowCompileInput) (explicit-compile-input-required).
  • Workflow compilation requires ProtocolStatus.Approved; other statuses are rejected (invalid-protocol-status).
  • Template and protocol identity/digests are validated and enforced; stale values are rejected.
  • Schema closure is enforced with repository-known schema refs; missing/unknown schema ids are hard failures.
  • Deterministic ordering is enforced for nodes, edges, approvals, capabilities, artifacts, resolved bindings, and invalidation entries.

Evidence: GATE-04-EVIDENCE.

Current issues to track

  • Execution semantics are still outside scope. Gate 4 documents compile output only; no workflow engine, provider calls, artifact storage, or persistence is implemented here.
  • Blueprint/protocol compatibility remains a non-claim. Negative/positive fixtures explicitly enforce no-blueprint-conformance-claim and no-php-compatibility-claim.
  • Gate-5 and beyond are pending. Workflow execution records and governance runtime behavior are handled in later gates.
