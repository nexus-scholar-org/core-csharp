Module documentation

Workflow

Deterministic workflow compilation from approved protocol versions into reproducible planned graphs with explicit gate semantics, waiver controls, capability requirements, and invalidation planning.

Scope and ownership

Workflow is a domain module that translates approved protocol content into a planned graph for controlled scientific conduct. It does not execute tasks, persist workflows, host plugins, or call providers. It only composes planning artifacts that other modules (CLI, bundles, provenance, UI) consume.

  • Primary input: WorkflowCompileInput (protocol + template + compile parameters + known schema refs + optional amendments + notices + compiler id/version).
  • Primary output: WorkflowDefinition (workflow id, workflow digest, resolved input bindings, topologically-ordered nodes/edges, approval requirements, capability requirements, artifact declarations, invalidation plan).
  • Domain ownership: all canonical graph/digest decisions remain here. Runtime execution, storage, and plugin capability grants are handled outside Gate 4.
  • Authority boundary: protocol approval is authoritative for protocol versions; workflow approval is compiled planning metadata only.

Core types and contracts

  • WorkflowTemplate: identity (template_id, template_version), digests, schema id/version, inputs, nodes, edges, gates, approval/capability/waiver/invalidation policy sets.
  • WorkflowTemplateInput with ScientificConduct or ExecutionParameter mode and optional default values.
  • WorkflowTemplateNode with kind/mode, dependencies, produced artifacts, optional approval / waiver / invalidation references.
  • WorkflowCompiledNode and WorkflowCompiledEdge represent graph structure after validation and topological ordering.
  • WorkflowResolvedInputBinding records exact source authority for each required input, including source type and digests.
  • WorkflowInvalidationPlanEntry records amendment / notice binding for impacted workflow nodes and artifacts.
  • WorkflowRuleException carries stable categories (e.g. invalid-protocol-status, unknown-node-requirement, invalid-hybrid-node).

Gate and ADR status

GATE-04 is implemented for local deterministic workflow compiler scope. Core decisions are ADR 0005 and ADR 0006.

  • Compile is explicit-only: WorkflowCompiler.Compile(ProtocolVersion) must not be used; callers must pass WorkflowCompileInput.
  • Protocol status contract: only ProtocolStatus.Approved is accepted by default compiler mode.
  • Compile is explicit-only: WorkflowCompiler.Compile(ProtocolVersion) is rejected in favor of Compile(WorkflowCompileInput) (explicit-compile-input-required).
  • Workflow compilation requires ProtocolStatus.Approved; other statuses are rejected (invalid-protocol-status).
  • Template and protocol digests are recomputed during compilation and must match supplied digests (stale-template-digest, stale-protocol-digest).
  • Template schema closure is mandatory: missing schema id/version or unknown schema pair fails locally.
  • Deterministic outputs: graph order, bindings, approvals, capabilities, artifacts, and invalidation entries are normalized before digesting.

Evidence: GATE-04-EVIDENCE.

Validation pipeline

  • Input normalization: template nodes are normalized for null-safe arrays and id trimming/validation.
  • Amendment witness check: amended protocol versions require matching amendment source in compile input.
  • Protocol digest validation: prevents accepting stale protocol material.
  • Template schema closure: validates all schema references against known refs supplied in input.
  • Template digest validation: digest mismatch stops compilation.
  • Template graph validation:
    • duplicate node ids
    • edges reference known nodes
    • no self-edges
    • topological sorting without cycles
  • Artifact checks:
    • all produced artifacts must be declared
    • all declarations must name an existing producer node
    • artifact declarations are carried into compiled output
  • Role/authority checks:
    • approval roles exist in template role registry
    • human-only approval policy is required (allows_automation=false)
    • approval gating/decision references are known
  • Node requirements:
    • node requirement refs resolve to template approvals, waivable items, or invalidation policy refs
    • hybrid nodes require capability refs and an explicit human approval/review path
  • Input resolution:
    • scientific conduct inputs resolve only from protocol decisions/waivers, never from compile parameters
    • execution parameters may resolve from explicit parameters or template defaults
    • unknown compile parameter keys are rejected
  • Waiver validation:
    • waiver must map to required template requirements
    • disclosure and consequence warning required
    • must not be expired relative to approved protocol time
    • must bind to approval identity
  • Invalidation planning:
    • requires amendment + notices for amended versions
    • rejects stale, unknown, or mismatched notices
    • rejects affected artifact digest mismatch and unknown node ids

Digest and identity contracts

  • workflow_id format: workflow-{first16lowercasehexsha256(material)}.
  • Workflow id material includes protocol id/version + content digest + template id/version + compiler id/version.
  • workflow_digest scope: canonical-json-record, schema nexus.workflow-definition, version 1.0.0.
  • Template digest scope: canonical-json-record, schema nexus.workflow-template, version 1.0.0.
  • Digest exclusions include execution records, provenance events, bundle manifests, provider responses, machine-local file paths, and caches.

Known issue taxonomy currently covered

  • No runtime execution claims: this gate does not execute tasks, emit UI state, or persist execution events.
  • No PHP/blueprint compatibility claim: explicit comparison rules and fixture metadata keep this local-scoped.
  • No AI execution claim: AI/task references in templates remain capability declarations only.

Current issues to track

  • Execution engine absent by design. Gate 4 scope is planning-only, no runtime planner runner.
  • Compatibility boundary explicit. Fixture comparison rules state no PHP compatibility and no blueprint conformance claims.
  • Runtime governance not yet integrated. Execution-level governance and audit append behavior remain future gates.
